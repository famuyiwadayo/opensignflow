# Data Model Draft

## Purpose

This document defines the first-pass database model for OpenSignFlow.

The goal is to make the schema strong enough for a serious SaaS-style product while keeping the first MVP achievable.

## Key modeling decisions

### 1. Use organizations/workspaces from day one

Even if the first UI only supports one personal workspace per user, documents, billing, usage, and audit logs should be scoped to an organization/workspace.

Why:

- team accounts will be easier to add later;
- billing belongs more naturally to a workspace than to a single user;
- document permissions become clearer;
- the product looks more production-ready.

MVP simplification:

- registration creates a default personal organization automatically;
- team invitations and organization settings can be hidden until later;
- most users will never notice the organization layer in v1.

### 2. Use opaque string IDs

The API should expose opaque string IDs such as:

```txt
usr_...
org_...
doc_...
fld_...
rcp_...
aud_...
job_...
```

IDs should be generated by the application, not by sequential database counters.

### 3. Store files privately

The database stores file metadata and private storage keys. It should not store public permanent URLs.

Examples:

```txt
originalStorageKey
completedStorageKey
signatureStorageKey
```

The API generates short-lived signed URLs when the user needs to preview or download a file.

### 4. Never store raw signing tokens

Public signing links contain high-entropy tokens. The database should store only token hashes.

Flow:

```txt
raw token in email link
        ↓
backend hashes token on request
        ↓
compare with stored tokenHash
```

### 5. Audit sensitive actions

Document-related actions should create audit events:

- upload;
- field changes;
- recipient changes;
- send;
- signing link opened;
- recipient viewed;
- recipient signed;
- final PDF generated;
- AI analysis generated.

### 6. Normalize PDF field coordinates

Field positions use normalized coordinates from `0` to `1`.

- `pageNumber` starts at `1`.
- `x` and `y` are top-left viewer coordinates.
- `width` and `height` are relative to page size.
- final PDF generation converts these into PDF coordinate space.

## Status transitions

### Document

```txt
DRAFT
  -> SENT
  -> VIEWED
  -> PARTIALLY_SIGNED
  -> COMPLETED

DRAFT/SENT/VIEWED/PARTIALLY_SIGNED
  -> CANCELLED
```

Rules:

- fields and recipients are editable only while `DRAFT`;
- sending requires at least one recipient and one required field;
- finalization runs after all required recipients sign;
- completed documents should not be modified.

### Recipient

```txt
PENDING -> SENT -> VIEWED -> SIGNED
PENDING/SENT/VIEWED -> DECLINED
PENDING/SENT/VIEWED -> EXPIRED
```

- users;
- personal organization creation;
- documents;
- recipients;
- fields;
- signing requests;
- signing submissions;
- field values;
